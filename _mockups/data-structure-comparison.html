Data Structure Options: Front/Back Views

Option A: Simple dual-field

- name: "Phayno (Chasuble)"
  image: "/assets/img/sacred-items/phayno_medium.png"
  back_image: "/assets/img/sacred-items/phayno_back_medium.png"
  malayalam: "ഫായ്നോ"
  meaning: "The robe of glory..."
✓ Simple — Easy to understand, minimal new fields
✓ Minimal changes — Backward compatible, existing items unaffected
✓ Clear intent — Self-documenting: front vs back is obvious
✗ Limited — Hard to add 3rd view in future (would need back_image_2, etc.)
✗ No labels — Can't customize "back" label (could be "rear," "side view," etc.)

Option B: Flexible images array

- name: "Phayno (Chasuble)"
  images:
    - view: "front"
      image: "/assets/img/sacred-items/phayno_medium.png"
    - view: "back"
      image: "/assets/img/sacred-items/phayno_back_medium.png"
  malayalam: "ഫായ്നോ"
  meaning: "The robe of glory..."
✓ Future-proof — Easy to add 3rd, 4th view later
✓ Flexible labels — Can use any view name ("side," "detail," etc.)
✓ Scalable — Works for items with many views
✗ More complex — Nested structure, harder to read at a glance
✗ Refactoring needed — Existing single-image items need updating
✗ Breaking change — Frontend code must handle both old and new formats

Which items would benefit from front/back views?

Vestments: Phayno (Chasuble), potentially others
Complex objects: Dhoopakutty (censer) — could show front assembly and internal detail
Future items: Any sacred item with multiple visual aspects
Recommendation: Start with Option A (simple dual-field). If/when you need 3+ views for multiple items, refactor to Option B. This follows YAGNI principle—don't over-engineer until you have concrete requirements.
